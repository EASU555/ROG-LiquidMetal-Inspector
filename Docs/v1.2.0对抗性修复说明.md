# v1.2.0 对抗性修复说明

v1.2.2 补充修复：NVIDIA 限制原因改为每 5 秒后台刷新并使用缓存，CPU/GPU 关键温度采样不再同步等待 `nvidia-smi`；5 秒 LibreHardwareMonitor 安全超时仍保留。界面、CSV 与 HTML 报告新增单次、平均和最大传感器读取耗时，方便区分真实传感器卡死与外部遥测延迟。

本版本修复除“管理员程序从可修改目录加载 DLL”之外的对抗性审查问题。该 DLL 加载项按用户要求保留现状，发布目录必须来自可信来源并核对 `checksums.sha256`。

## 防止错误绿色

- 零负载现在是有效的 0%，不会被正数过滤器丢弃。
- 分别记录 CPU/GPU 负载字段覆盖率；负载字段缺失、平均负载不足或波动过大时不能判正常。
- 按采样间隔计算应有样本数，并校验最大间隔；两个相隔五分钟的样本不会再伪装成完整五分钟测试。
- 传感器单次读取超过 5 秒即停止压力并令测试无效；传感器关闭也有 3 秒等待上限。
- 温度趋势超过稳态边界、当前性能档位没有对应机型参考、或档案要求的测试条件未确认时，只能输出建议复测。

## 防止错误红色

- 同一组件同一因果类别只取最高分，功耗下降和频率下降不再重复累加。
- 红色结论除两类证据外，还必须包含空间热点、驱动确认的热限制或本机历史恶化之一。
- 通过 `nvidia-smi` 的绝对系统路径读取热限制、功耗上限和 Power Brake 原因。驱动已报告功耗/供电限制时，不把同一次功耗/频率下降再次归因于散热接触。
- GPU 功耗低于档案但温度不高只记录配置差异，异常分为 0。

## 复现、基线与设备身份

- 本地 `History/history.json` 保存最多 100 次完整测试摘要；同条件定义为同机型、CPU、GPU、模块、档位、规则版本、档案哈希且室温相差不超过 2°C。
- 正常或异常结果需最近三次中至少两次复现；第一次结果会显示“等待重复确认”。
- 至少两次健康记录后，以每瓦温升中位数建立本机基线；当前值恶化 15% 才形成纵向证据。
- 热点探测阶段现在用于验证主测固定热点是否跨阶段复现。
- OpenCL 压力设备名称必须与传感器采样 GPU 一致，否则立即中止。

## 配置、报告与构建

- 规则字段增加严格上下限；空匹配字段的机型档案不能匹配所有机器，档案数值也会先校验。
- 报告新增时间轴覆盖、最大缺口、规则/档案哈希、GPU 驱动与 PCI Bus ID、限制原因、风扇分类、每瓦温升、历史复现和基线状态。
- CPU、系统和 GPU 风扇分开记录，GPU 风扇不会再写入 CPU 风扇字段。
- 构建下载 LibreHardwareMonitor v0.9.6 时强制校验 SHA-256，输出依赖清单、逐文件校验清单和 ZIP 校验文件，并支持可选 Authenticode 签名。
- v1.2.1 自动测试从 10 项增加到 19 项，覆盖稀疏采样、零负载、温度爬升、档位不匹配、双烤因果重复、重复运行门槛、恶意规则值、空档案匹配和默认规则误报警告。
